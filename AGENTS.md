# AGENTS.md — IBKR-Telegram

## Project Overview

**IBKR-Telegram** is a self-hosted Telegram bot for Interactive Brokers. It provides portfolio management, trade execution, and position monitoring via Telegram commands. Designed for multi-account setups with options-first trading (LEAPS), safety checks, and margin compliance.

## Tech Stack

- **Language**: Python 3.12+, fully async (`asyncio`)
- **Telegram**: `aiogram` 3.x (bot framework, inline keyboards, callback queries)
- **IBKR**: `ib-async` (async wrapper for TWS/Gateway API)
- **Database**: SQLite via `aiosqlite` (trade log, signal history, deposits)
- **Config**: YAML (`pyyaml`) with environment variable overrides
- **Docker**: `docker` SDK for gateway container management (pause/resume)
- **XML**: `defusedxml` for secure parsing of IBKR Flex Web Service responses
- **Testing**: `pytest` + `pytest-asyncio`
- **Linting**: `ruff`

## Project Structure

```
src/
  __init__.py       # Package docstring
  __main__.py       # Entry point (logging setup, asyncio.run)
  app.py            # Orchestration: wires bot, executor, DB, periodic sync
  bot.py            # Telegram command handlers, inline keyboards, confirmation flows
  config.py         # YAML + env var config loading, dataclass validation
  db.py             # SQLite schema, trade/signal/deposit CRUD
  executor.py       # IBKR connection, order sizing, option chain, margin checks
  models.py         # TradeSignal dataclass (shared across modules)
  safety.py         # Market hours, position limits, duplicate detection
tests/
  test_executor.py  # Sell fraction parsing, ExecutionResult
  test_safety.py    # Market hours, position limits, duplicate detection
```

## Coding Conventions

- **Line length**: 120 characters (`ruff` enforced)
- **Imports**: sorted by `isort` rules via `ruff` (stdlib, third-party, local)
- **Type hints**: use throughout; `X | None` syntax (not `Optional[X]`)
- **Async**: all I/O operations are async. Never use blocking calls in the event loop.
- **Logging**: use `logging.getLogger(__name__)` per module. Logger name for the app is `ibkr_telegram`.
- **Error handling**: IBKR operations must handle `ConnectionError`, `TimeoutError`, and `asyncio.TimeoutError`. Financial calculations must guard against `NaN`, zero prices, and missing data.
- **SQL**: all queries use parameterized placeholders (`?`), never string interpolation.
- **Config**: environment variables override YAML values. Per-account overrides use `MARGIN_MODE_<NAME>` and `MAX_MARGIN_<NAME>` patterns.

## Key Design Decisions

- **TradeSignal in models.py**: extracted into its own module to avoid circular imports. All modules import from `models`, not from each other.
- **Multi-account**: `TradeExecutor` maintains one `IBKRConnector` per configured account. Orders execute across all accounts with the same percentage allocation.
- **Confirmation flow**: every trade goes through a Telegram inline keyboard confirmation before execution (`confirm_before_execute` config).
- **Option chain resolution**: for LEAPS, selects the furthest expiry with the deepest ITM strike. Uses `reqSecDefOptParams` + `qualifyContracts` from ib-async.
- **Margin modes**: `off` (cash only), `soft` (margin-aware sizing + alerts), `hard` (soft + auto-sell enforcement). Configured per account.
- **Gateway containers**: derived from `config.accounts[].gateway_host`. The bot can pause/resume these Docker containers for manual IBKR login.

## Running

```bash
# Development
pip install -e ".[dev]"
python -m src

# Tests
pytest tests/ -v

# Lint
ruff check .

# Docker
docker build -t ibkr-telegram .
docker compose -f docker-compose.example.yml up
```
